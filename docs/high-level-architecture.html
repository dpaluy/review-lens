System Context

What The Application Is

ReviewLens AI is intentionally narrow. It is not a generic chatbot or crawler. The demo-critical path is one product corpus, captured from Trustpilot or manual import, then queried through a product-scoped grounded Q&A service.

Analyst Pastes a Trustpilot review URL or pasted review blocks, monitors ingestion, asks product questions.
->
ReviewLens Rails App Validates source, fetches or parses review content, stores normalized reviews, runs summaries, answers questions.
->
External Services Trustpilot public pages for URL ingestion, OpenAI through RubyLLM for structured AI calls, DigitalOcean for hosting.
Runtime Architecture

Rails Monolith, Separate Responsibilities

The app keeps operational complexity low: Rails handles web UI, background work, Active Record persistence, and Turbo updates. The important separation is at service boundaries, not at process or microservice boundaries.

Critical Flows

How Data Moves

Both ingestion paths converge on normalized reviews. Q&A then consumes persisted summaries and selected review text, not live web pages or generic model memory.

URL Ingestion Flow

  1. 1Create productPOST /products validates Trustpilot URL and creates an ingestion run.
  2. 2Fetch public HTMLIngestion::Fetcher allows only Trustpilot hosts, two redirects, ten seconds, five MB.
  3. 3Probe and parseTrustpilotProbe detects thin corpus or blocking, then TrustpilotAdapter extracts review cards.
  4. 4Normalize reviewsIngestion::Importer truncates long bodies, deduplicates by external ID or content hash, updates product summary.
  5. 5Summarize corpusBatchSummarizer stores structured 30-review insight batches and marks product ready.

Grounded Q&A Flow

  1. 1Ask from productProduct chat creates a product-linked Conversation and enqueues response generation.
  2. 2Scope firstScopeGuard blocks weather, competitor comparisons, other review sources, and outside knowledge.
  3. 3Build contextContextBuilder combines product summary, insight batches, and up to 50 keyword-matched reviews.
  4. 4Answer with schemaQuestionAnswerer asks RubyLLM for answer markdown, confidence, limitations, and review IDs.
  5. 5Persist evidenceAssistant response stores raw metadata in chat_messages.content_raw.
Application Components

Main Code Boundaries

The useful architecture line is between entry points, ingestion services, review analysis services, and persistence. Avoiding extra layers is a strength here.

Product Entry Points

ProductsController owns new, create, and show. URL imports enqueue a job. Manual imports run through a service inside the request transaction.

Review Platform Adapters

TrustpilotAdapter parses public HTML. ManualAdapter parses pasted text or CSV-like input. Network fetching stays outside adapters.

Ingestion Pipeline

Fetcher, TrustpilotProbe, Importer, and SummaryBuilder enforce host limits, quality checks, dedupe, and product-level summary data.

Review Analysis

BatchSummarizer creates durable AI summaries. ContextBuilder selects bounded review evidence for each question.

Guarded Answering

ScopeGuard classifies scope before generation. QuestionAnswerer refuses blocked prompts or returns schema-bound answer metadata.

Generic Chat Foundation

/conversations also supports generic RubyLLM chat. That is separate from the product-scoped demo path and should not be confused with guarded review Q&A.

Data Architecture

Tables That Matter

PostgreSQL is the system of record. The key design choice is that review evidence, AI summaries, refusals, and answer metadata are persisted for reviewability.

Table Purpose Important Fields Important Constraints
products One ingested review corpus and its ingestion state. platform, external_id, source_url, ingestion_status, reviews_count, rating_distribution, ingestion_summary. Unique platform and external_id. Status indexed.
ingestion_runs Execution record for each import attempt. status, parser_version, raw_fetch_metadata, warnings, reviews_found, reviews_imported, reviews_skipped, error. Belongs to product. Status indexed.
reviews Normalized evidence corpus. body, title, rating, sentiment, reviewer metadata, review_date, raw_payload, content_hash. Unique product plus content_hash, unique product plus external_review_id when present.
insight_batches Structured AI summaries over review batches. batch_index, review_ids, reviews_count, summary JSON. Unique product plus batch_index.
conversations Chat session, optionally tied to a product. product_id, ai_model_id, timestamps. Product is optional, enabling generic chat and product-scoped chat.
chat_messages User and assistant messages with answer metadata. role, content, content_raw, token fields, tool_call_id. Conversation required. Role indexed.
AI Guardrails

Evidence First, Refuse The Rest

The product path uses three AI-adjacent stages: batch summarization, scope classification, and answer generation. The guardrail runs before answer generation.

ScopeGuard Deterministic pattern checks plus model classification reject other platforms, competitor comparisons, current facts, and external review sources.
->
QuestionAnswerer Uses only supplied context. Returns answer markdown, confidence, supporting review IDs, limitations, or refusal metadata.
Architecture constraint Product answers must be grounded in stored review context. Live browsing, broad model knowledge, competitor claims, and cross-platform review comparisons are outside the product contract.

Architecture Notes

Known Risks And Honest Boundaries

These are not reasons to add architecture. They are the places where a reviewer will look for proof that the current simple architecture is deliberate.

Trustpilot Is Best Effort

The fetcher refuses unsupported hosts, large responses, excessive redirects, and blocking pages. If public HTML is thin or blocked, manual import is the correct fallback.

README Is Behind Code

The README still says grounded Q&A needs layering. Current services now include product-scoped scope guard, context builder, and question answerer, so docs should be refreshed before final review.

Generic Chat Is A Side Path

The generic RubyLLM conversation route is useful foundation code, but demo evaluation should use product-linked conversations so guardrails and review evidence are enforced.

No Vector Store By Design

Context selection uses summaries plus keyword-matched reviews capped at 50. That is enough for the exercise and avoids premature RAG infrastructure.
